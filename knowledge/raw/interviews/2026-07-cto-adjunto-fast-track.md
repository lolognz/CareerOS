## Professional Philosophy

Architecture:
- The user thinks better with pen and paper.
- Before opening an IDE, the user designs the idea at the highest useful abstraction level.
- The user sketches front, backend, database, deployment, monolith or microservices, system boundaries and then drills down.
- Documentation can start from this design.
- The user is most comfortable with backend and databases.
- The user would delegate frontend to someone stronger when possible.
- In backend, the user thinks about layers, communication with frontend and database, synchronous/asynchronous/mixed systems and necessary components.
- Depending on project complexity, a lot of time may pass before opening the IDE.

Microservices vs monolith:
- The user uses two main variables:
  - Expected traffic.
  - Whether the domain has clear independent service boundaries.
- If blocks can work independently, microservices make sense.
- If not, monolith is preferred.
- High traffic increases interest in finding a microservice split.
- Low-traffic applications do not need unnecessary distributed complexity.

Good architecture:
- Easy to understand.
- Easy to scale.
- Divides responsibilities into small understandable packages.
- Each package does one task.
- Example from Mercury: each document type had its own microservice, so traffic did not disturb others and the service purpose was clear.

Bad architecture signal:
- One component does many different things and receives high traffic.
- This creates a maintenance tangle likely to break as features grow.

Code:
- Clean Code means descriptive class, function and variable names.
- Classes should do one thing.
- Methods should be simple.
- Useful comments should be added where they speed up understanding.
- Comments are not always needed for obvious one- or two-line code.
- Unit and integration tests matter.
- A project should compile with one button and start with another.

Working with existing code:
- The user first looks for documentation and Postman/Bruno test collections.
- The user tries to understand structure and communication before looking deeply into code.
- With many microservices, some services may only need dependency upgrades or Java migrations without opening much business code.

Requirements:
- Ambiguous requirements must be clarified.
- If clarification is not possible upfront, the user prefers small developments validated with business rather than building everything based on assumptions.
- The user asks when something comes from outside, when the requirement is unclear or when blocked.
- The user decides technical implementation details when the business result is the same.

Ready to code:
- The user needs to understand at least one complete flow.
- Not all use cases are needed upfront.
- A happy path or complete technical slice is enough to start.
- Example: if a microservice will use MQ, configure and test MQ first, then build the flow for reading from the queue.

Quality:
- The user never wants to deploy something not heavily tested.
- In critical teams, incidents can create high stress and out-of-hours work.
- Before merge, the user reviews code, verifies tests and asks the PR author to confirm local testing.
- In senior teams, the user trusts colleagues' word.
- With juniors, the user would likely download the branch, run locally and test.
- Tests are considered essential.

Team:
- When a junior joins, the first priority is making them comfortable.
- People work better when comfortable in the team.
- They should feel part of jokes, safe to speak, and safe to make mistakes.
- If a colleague submits bad code, the user usually asks what it is supposed to do and suggests a clearer alternative.
- With enough trust, the user is more direct and asks for specific improvements.
- Technical disagreement should be debated with all options on the table until the best option emerges.
- Teamwork means everyone works in the same direction.
- If someone makes a mistake, the team supports them.
- Anyone should be able to ask questions and ask for help.
- The work environment should be pleasant, relaxed and professional.
